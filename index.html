Internal Prototype. This dashboard is an internal research prototype developed under NC Sea Grant funding. It is intended for team review and stakeholder discussion only and is not approved for public distribution.
Study Area and Network

The analysis covers the southeastern North Carolina coastal plain, spanning the Interstate 40 corridor from Fayetteville to Wilmington across Cumberland, Bladen, Columbus, Robeson, Sampson, Duplin, Pender, Brunswick, and New Hanover counties. The road network was extracted from OpenStreetMap and comprises 115,760 intersection nodes and 279,485 directed road segments.

Data Preparation Methods
1
POI Classification
All 59,289 SafeGraph POIs were classified using NAICS codes into three essential tiers. Tier 1 covers life safety facilities including hospitals, clinics, and pharmacies. Tier 2 covers emergency operations support including grocery stores, hotels, and utilities. Tier 3 covers vulnerable population access including social services and child care. POIs not meeting any tier criterion are excluded from emergency access scoring but retained in the dataset for visualization.
2
POI Visit Count Correction
SafeGraph assigns visit counts to individual POI records. When a building contains multiple listed tenants and the data cannot determine which specific tenant was visited, the same count is assigned to all co-located records. To correct for this, all 14,650 retained POIs were grouped using a standardized address key built from street address, city, and postal code. Of the 14,650 POIs, 6,032 shared their address with at least one other POI across 2,261 unique shared addresses. Two duplication patterns were identified: (1) exact duplication, where every co-located POI carried an identical visit count - for example, 13 providers at one Wilmington address each showing 4,155 visits, inflating the apparent total to 54,015; and (2) partial duplication in multi-provider complexes, where some providers shared the top count and others had distinct values. Within each shared-address group, the maximum observed 2024 visit count was retained as the representative demand signal.
3
NCEM Classification
NCEM features were classified using occupancy type codes from the HAZUS 6.1 Flood Model Technical Manual. Features are marked eligible for scoring if their occupancy type corresponds to emergency operations, essential services, or critical infrastructure under HAZUS 6.1 definitions. Agricultural buildings are excluded from scoring due to their high density in rural areas, which would introduce geographic bias into node scores unrelated to life-safety function.
4
AADT Conflation

The AADT conflation process transfers NCDOT traffic count values onto the OSM road network. This was necessary because NCDOT AADT segments and OSM road edges do not share a common road identifier.

  • Both NCDOT AADT segments and OSM roads were projected to UTM Zone 17N to support meter-based distance calculations.
  • The statewide NCDOT AADT layer contained 47,735 road segments. It was clipped to the TRIP study area with a 0.05° buffer to retain roads crossing the boundary. This reduced the working dataset to 12,374 segments.
  • NCDOT road segments were reconstructed by grouping records with the same RouteID and ordering them by beginning milepost. This created continuous route geometries for matching.
  • A 50 m gap check was applied during route reconstruction. Adjacent segments separated by more than 50 m were treated as separate route parts to avoid false connections. This identified 1,013 route breaks across 5,420 routes.
  • Reconstructed routes were used only to generate search points. Final AADT values were transferred from the original NCDOT segments to preserve traffic variation along each route.
  • Dense points were generated every 10 m along the reconstructed NCDOT routes. This reduced missed matches on long, curved, or irregular road segments.
  • Only matchable OSM road classes were retained: motorway, trunk, primary, secondary, tertiary, and their link variants. This produced 58,375 eligible OSM edges.
  • A KD-tree nearest-neighbor search identified candidate NCDOT route points within 100 m of each OSM road point.
  • A 30° bearing filter removed candidates with poor directional agreement. This helped avoid false matches between parallel roads, service roads, ramps, and nearby unrelated links.
  • When multiple candidates remained, Hausdorff distance was used to select the best matching NCDOT segment. This compares the overall shape alignment of two road geometries, not just one point location.
  • Each matched OSM edge received the AADT value from its best matching NCDOT segment.
  • A neighbor consistency check flagged OSM edges whose matched RouteID differed from surrounding connected edges. These isolated cases were marked for review.
  • The final match rate was 65.7 percent, with 38,340 of 58,375 eligible OSM edges matched to NCDOT AADT data.
  • The median Hausdorff distance was 2.9 m, indicating strong geometric agreement for most matched roads.
  • Neighbor consistency was high, with 97.8 percent of matched edges agreeing with surrounding connected edges. A total of 685 isolated matches were flagged for review.
5
Facility-to-Network Embedding
Each facility was assigned to the road network using an edge-based projection procedure. For each facility, the nearest driveable road edge was identified using a spatial index query with expanding search radii (200 m, 500 m, 1,000 m, 2,000 m), followed by true point-to-geometry distance computation. The facility point was then projected perpendicularly onto that edge. Both endpoint nodes of the matched edge received the same weight of 1, reflecting that access direction is unknown and either endpoint independently enables complete access to the facility. Polygon geometries such as NCDOT land parcels were converted to centroids before snapping.
Node Criticality Scoring

Every OSM road network node is scored across two independent dimensions. Storing dimensions independently allows composite weights to be adjusted in the dashboard without rerunning the spatial analysis.

Emergency Access (80%). A weighted sum of facility counts assigned to each node. Because the embedding procedure produces two records per facility, counts are deduplicated on the combination of facility identifier, node identifier, and data source before aggregation. The weights reflect relative life-safety priority during a flood event:

Composite score: (emergency score × 0.80) + (demand score × 0.20).

Gateway and Chokepoint Analysis

The node criticality analysis identifies locations that are important because of the facilities located near them, such as hospitals, emergency services, government facilities, and other essential destinations. The Gateway and Chokepoint Analysis provides a complementary perspective by identifying locations that are important because of the access they provide to those destinations.

A node may have no facilities directly assigned to it and still be important. Some nodes act as access-support locations that help connect communities to hospitals, emergency facilities, shelters, and other Critical destinations. Disruption at these locations could affect access to important services across the transportation network.

The analysis used 814 Critical destinations consisting of all 787 Critical-tier nodes identified under the default scoring configuration, together with 27 hospital nodes that scored below the Critical threshold. The destination set is fixed to the default scoring configuration.

1
Critical Destination Access Support
For each of the 814 Critical destinations, Dijkstra's shortest path algorithm was run on the reversed transportation network with a 50 km search distance. This identifies the road nodes that contribute to access to each Critical destination. Nodes receive higher scores when they repeatedly appear across access routes serving multiple Critical destinations. The resulting metric highlights locations that support connectivity to essential facilities throughout the study area.
2
Destination-Specific Structural Chokepoint Screening
For each Critical destination, the subgraph of all nodes that can reach that destination within 50 km was extracted. Tarjan's depth-first search articulation point algorithm was then applied to each destination-specific subgraph. Nodes identified as articulation points represent locations whose removal would disconnect portions of the reachable network, causing some origins to lose road access to that specific destination. Unlike traditional network-wide topology measures, this screening is performed separately for each Critical destination and therefore identifies destination-specific chokepoints.
3
Targeted Exact Node-Deletion Validation
The top 250 gateway nodes were validated by exact testing. For each candidate node, the node was temporarily removed from the network and Dijkstra's shortest path algorithm was rerun. Nodes whose removal caused one or more origins to lose all access to a Critical destination were classified as Validated Hard Chokepoints. Nodes that remained reachable but whose travel distance increased by 10% or more were classified as Soft Bottlenecks. Of the 250 candidates tested, 174 were Validated Hard Chokepoints.

Planned Methodology Improvements
POI Deduplication
Polygon Footprint Overlap Detection
The current deduplication uses normalized address keys to group co-located POIs. A planned improvement uses the polygon_wkt field from SafeGraph - the actual building footprint polygon for each POI - to detect spatial overlap between footprints directly. Two POIs whose footprint polygons overlap or share the same building boundary are co-located regardless of how their addresses are formatted. This approach is more robust than address matching for cases where address normalization fails due to formatting inconsistencies, and it can detect co-location across addresses that share a building but use different street number formats.
Network
Full-Geometry Network Download
The current network uses a simplified OSMnx graph retaining intersection nodes only. A planned improvement is to test simplify=False so all road geometry nodes are retained. This may improve facility-to-network embedding for facilities located on mid-block road segments. Computation time and memory use are also important constraints, so this change will be evaluated before it is adopted for the full regional workflow.
AADT Conflation
Conflation Validation and RoadID Dictionary
A visual validation layer and OSMID-to-NCDOT RoadID dictionary will allow matched AADT values to be verified and traced to source records.
Gateway Analysis
Expanded Validation Strategies
The current exact validation covers the top 250 gateway nodes because exact node-deletion testing is computationally expensive at regional scale. Applying exact validation to every candidate may not be feasible. Alternative validation approaches are being explored to improve coverage while keeping computation time practical.
